# TD mode / dashboard / registry gotchas (Stage 3aâ€“3d)

## Stage 3a engine contract gotchas

- **`PROCESSED` events are capability-emitted only when opted in.** The engine declares `"PROCESSED"` in `RequestEventType` but never emits one. Production capabilities emit PROCESSED only when constructed with `emitProcessedEvent: true`. TD integration tests depend on this via `buildServer` / `buildDatabase` options. Sandbox/dashboard usage (no options) gets no PROCESSED events.
- **`FORWARDED` events come from two sources.** The engine's `deliverStaged` emits a `FORWARDED` at the **target** component with `capabilityId: null`. `ForwardingCapability` emits its own `FORWARDED` at the **source** component with `capabilityId: this.id` **when constructed with `emitForwardedEvent: true`**. Integration test helpers filter `ev.capabilityId !== null` to count source-side forwards.
- **PASS at PROCESS phase is converted to DROP with reason `no_handler`** (Stage 3c change, was a silent drop in Stage 3a/3b). If no PROCESS capability matched `canHandle(request.type)` â€” or the matching cap explicitly returned PASS â€” `Component.process` now returns `{kind: "DROP", reason: "no_handler"}` before REPLICATE/OBSERVE run. `deliverStaged` hits the DROP case: emits a `DROPPED` event at the component, increments `perComponentThisTick.drops`, applies cascade. `ForwardingCapability` is still the explicit primitive that produces FORWARD â€” "let it fall through to egress" is still not a pipeline behavior. The Stage 3a Server/Forwarding write-routing pattern is unchanged because the Forwarding cap's `canHandle` returns true for writes, so the trailing-PASS fallback never triggers for that path.
- **`componentThroughputPerTick` sums all PROCESS capabilities whose `getThroughputPerTick` is defined.** A Server with `ProcessingCapability(throughputPerTier=20)` + `ForwardingCapability(throughputPerTier=12)` has a total budget of 32 req/tick (at T1). An intermediary LB/Gateway/CDN with `ForwardingCapability` (no throughput option) contributes nothing and the component has unbounded throughput â€” the sandbox/dashboard default.
- **Registry factories instantiate throwaway capabilities.** `ComponentRegistry.validate()` calls each capability's factory once to check phases and sub-interfaces. These instances don't run in simulation; per-tick components are constructed separately (via `registry.create(type, ...)` in sandbox, or via `buildX` harness helpers in TD tests). Registry-side defaults and per-component options can diverge.
- **`tryPlace` is real on TDModeController, throws on SandboxModeController.** TD's `tryPlace` was implemented in Stage 3b (registry mint â†’ budget check â†’ debit â†’ `state.placeComponent`). Sandbox's `tryPlace` and `tryUpgrade` were lying stubs that returned fake ids; the post-3b cleanup pass replaced both bodies with `throw new Error("not implemented yet")` until Stage 3c lands real impls. `TDModeController.tryUpgrade` was the same lie and now also throws. Sandbox topologies still place via `state.placeComponent()` directly through `src/dashboard/topologies.ts`.

## Stage 3b engine contract gotchas

- **`TDModeController` is multi-wave only.** The constructor takes `{waves, economy, entryPointId, rng, componentRegistry}`. Single-wave call sites (e.g. `tests/integration/td/helpers.ts:runWave`) pass `waves: [wave]` and thread a real `componentRegistry` via `bootTDRegistry()` from `@harness/td-fixtures`. The Stage 3b single-wave back-compat shim (`TDSingleWaveOptions` + `STUB_REGISTRY`) was deleted in Batch 3.
- **`advancePhase(state?)` snapshots `waveStartMetricsIndex` only when `state` is passed.** Stage 3a tests call `advancePhase()` no-arg and never read `getCurrentWaveMetrics`. Dashboard call sites pass `state`.
- **`registerTDDefaults` factories are TD-tuned.** `processing` registers with `{handledTypes: ["api_read"], throughputPerTier: 20, emitProcessedEvent: true}`. `forwarding` is Server-style writes-only at 12/tick. `forwarding-pipe` is the Cache/LB/Client variant at 55/tick. `storage` is at 25/tick with PROCESSED events. After Batch 3, `buildServer/buildDatabase/buildCache` in `tests/integration/td/helpers.ts` construct their components directly via `compRegistry.create(...)`, so these factories are the single source of truth for TD-mode component shape. `buildLoadBalancer` still hand-builds its component because `LOAD_BALANCER_ENTRY` has a fixed port shape and the wave tests need a variable egress count. Sandbox bootstrap (`bootstrapRegistries`) is unaffected.
- **`CLIENT_ENTRY` has no ingress port.** It's egress-only (entry point). `tryConnect(state, server, client)` rejects with `no_ingress_port`.
- **`isWaveDrained` walks four request locations:** `state.pending`, `state.blockedParents`, `state.activeStreams`, **and** `EngineBufferable.peekBuffered()` partitions on every component. None of TD's current capabilities are bufferable, but the primitive is reusable for future waves.
- **`TDEconomy.economy` is mutable on the controller.** The dashboard calls `tdController.setEconomy(...)` between waves to reset the budget. The Stage 3a `runWave` path constructs a fresh controller per wave so it never exercises this mutation.
- **Per-wave reset is dashboard-driven, not controller-driven.** The dashboard runs (1) `setEconomy(newEconomy)`, (2) `state.setCondition(id, 1.0)` for every component, (3) `advancePhase(state)` (assessâ†’build) on the wave boundary, AND (4) `state.recomputeVisitOrder()` to pick up freshly-placed components. The controller does not own this reset.
- **visitOrder refresh is `state.recomputeVisitOrder()`, not a fresh Engine.** Post-Batch-3, the Engine constructor delegates to `state.recomputeVisitOrder()` and call sites that need to update visitOrder mid-campaign (dashboard buildâ†’simulate, `campaign-headless.test.ts` per-wave handoff) call the state method directly and reuse the same `Engine` instance. `state.placeComponent` still does not auto-update visitOrder â€” that's a deliberate Stage 3c candidate.
- **Server `p-in` ingress port has `capacity: 1`.** Cache/LB rescue topologies that wanted to wire a second connection into an existing Server fail with `port_capacity_exceeded`. The `campaign-headless` test uses a second Server placement instead. Multi-port disambiguation is a Stage 3c topic.
- **`tryPlace` advances the registry's id counter even on rollback.** `ComponentRegistry.tryCreate` mints a `Component` (incrementing the internal counter) before `tryPlace` checks the budget. On `insufficient_budget` rejection, the component is discarded but the counter has advanced â€” `ComponentId` values may have gaps. No test depends on contiguous ids.
- **`tryConnect` uses first-matching-port.** First port with `direction === "egress"` on the source, first with `"ingress"` on the target. Components with multiple in-ports of different roles can't be disambiguated until Stage 3c.
- **`SimLoop` is now generic over `ModeController`.** Constructor takes an `onTick` callback that receives the concrete controller type, plus an optional `shouldStop` predicate. The sandbox call site passes a `SandboxModeController`-typed callback that calls `getMetricsSnapshot`; the TD call site passes its own. `SimLoop.reset(engine, state, controller)` swaps in a fresh engine without losing the callbacks.
- **TD HUD reads `tdController.economy.getBudget()`, not sandbox-only `totalRevenue`/`totalUpkeep`.** `TDEconomy` exposes only `getBudget()`. Cumulative wave revenue/upkeep, if shown, must come from `state.metricsHistory.revenueEarned`/`upkeepPaid` aggregation.

## Stage 3c engine contract gotchas

- **`state.lastTickEvents` is the per-tick event view for the renderer.** `state.appendEvent` writes through to both `requestLog` (existing, unbounded) and `lastTickEvents` (per-tick). `Engine.tick()` clears it as its first statement, before step 1. Consumers running AFTER `engine.tick()` returns see exactly this-tick's events.
- **`TickMetrics.perConnection` is optional** â€” existing tests that construct `TickMetrics` literals without it stay compiling. The Stage 3c Pixi adapter reads it; non-dashboard consumers ignore it.
- **FORWARDED events carry `metadata.requestType`.** Both the engine's target-side emit (`deliver-staged.ts`) and `ForwardingCapability`'s source-side emit (when `emitForwardedEvent: true`) tag their events with `{ requestType: request.type }`. Used by the renderer adapter to color per-request dots without maintaining a separate idâ†’type map.
- **`SERVER_ENTRY.p-in.capacity` is 2** (bumped from 1 in Task 1.4). Designed to enable single-Server Wave 3 cache-rescue â€” but the `campaign-headless.test.ts` test still uses the Stage-3b two-server rescue because the SLA gate (95% availability on Wave 3) rejects the single-server topology at ~92.5% availability. Single-server rewrite deferred: needs wave tuning (pool/intensity/SLA) in a follow-up pass before the test can flip.
- **`ComponentRegistryEntry` has optional `longDescription` + `capabilitiesHuman`.** TD entries populate them; the 14 sandbox entries in `src/core/registry/component-entries.ts` don't. Consumed by the TD info panel and briefing card.
- **Dashboard topology rendering is Pixi v8, not DOM.** `src/dashboard/render/pixi-topology-renderer.ts` owns the canvas. The dashboard depends on the `TopologyRenderer` interface in `src/dashboard/render/topology-renderer.ts`, not on `pixi.js` directly. `src/dashboard/render/state-to-renderer.ts` is the per-tick adapter that reads `TickMetrics` + `state.lastTickEvents` and drives component/connection/dot updates. `createTDDashboard` is now async because Pixi v8's `Application.init()` is async â€” `bootTDMode` awaits it, top-level call sites fire-and-forget via `void bootTDMode()`.
- **Engine-pixi isolation is a tested invariant.** `tests/unit/engine-pixi-isolation.test.ts` scans `src/core/**` and `src/capabilities/**` for `pixi.js` import specifiers. The test must stay green â€” the engine is framework-agnostic per the Phase 1 scope.
- **`rerenderTopology` on the Pixi dashboard is a diff-style re-seed.** It walks `state.components`/`state.connections` and adds/removes whatever doesn't match the renderer's tracked id sets. Called after the retry flow in `main.ts` replays actions against state directly (bypassing the renderer's pointerdown handler).
- **Info panel click vs connect flow:** clicking a component in TD build phase both opens the info panel AND initiates a connect (the component becomes the connect source). Click another component to complete the connection; click empty space to cancel.
- **Post-wave diagnosis lives in `src/dashboard/td/diagnose-wave.ts`.** Pure function, 4 branches ordered by specificity: write routing gap â†’ process throughput â†’ TTL timeouts â†’ default. Hints point at symptoms, never at solutions. Five-case unit test pins the branch order.
- **All dashboard DOM construction with dynamic content uses `textContent` + `createElement`, never `innerHTML`.** XSS defense; enforced culturally.
- **Pixi v8 API differences from v7 (for future renderer edits):** `await app.init({...})` is async; `Graphics` uses `g.rect(x,y,w,h).fill(color)` not `beginFill/drawRect`; `Text` is `new Text({ text, style })`.
- **`StorageCapability.canHandle` is TD-restricted to writes only.** New `handledTypes` option on `StorageCapabilityOptions` defaults to `["api_write", "api_read"]` (sandbox/legacy unchanged). `registerTDDefaults` passes `handledTypes: ["api_write"]` so a naked Clientâ†’Database cannot bypass the Server tier and trivially win Wave 1. Without this, `StorageCapability` was happily handling reads at 25/tick and the whole "Server is the read primitive" lesson collapsed.
- **Availability is measured against scheduled-generated, not accounted-for.** `evaluateSLA`, `evaluateOutcome`, and the `onTick` mid-wave penalty all use `max(accountedTotal, wave.intensity * wave.duration, 1)` as the availability/drop-rate denominator. Previously they used `resolved / (resolved + dropped + timedOut)`, which defaulted to `1.0` (WIN) when all three counters were zero â€” a vacuous-truth bug that let silent-PASS topologies trivially win. For fixed-intensity waves (all current ones) the scheduled count is exact; variable-intensity waves in the future will need TDTrafficSource to expose a counter.
- **`flashResponded` pulses green on every successful response.** Third one-shot flash alongside `flashDrop`/`flashOverload`. Dimmer and shorter (peakAlpha 0.35, 120ms, no padding) than the others because it fires on every served request â€” a Server handling 20/tick would strobe at the full flash intensity. Wired from `state-to-renderer.ts` on `RESPONDED` events.
- **`TDTrafficSource` generates ONE request type per tick.** Every request in a single `generate()` call shares the same `type`. The wave's `composition` distribution is realized across ticks, not within a single tick: `buildTypeSchedule(wave, rng)` uses largest-remainder rounding + Fisher-Yates shuffle to produce a `duration`-length array where the per-type counts match the composition ratio exactly. Wave 2 (30 ticks Ã— 70R/30W) has exactly 21 reads-ticks and 9 writes-ticks, shuffled. This was a visual fix â€” mixed per-tick traffic made the renderer draw reads+writes dots simultaneously on the same edge, reading as noise. One-type-per-tick makes each visible dot unambiguously "this tick is reads" or "this tick is writes."
- **Per-cap `getThroughputPerTick` contributions POOL into a single component budget.** `componentThroughputPerTick(comp)` sums every PROCESS-phase cap's contribution. It is NOT a type-segmented limit â€” a Server with `ProcessingCapability(reads, throughputPerTier: 15)` + `ForwardingCapability(writes, throughputPerTier: 15)` has a 30/tick pooled budget that can be spent on *any* 30 requests of any type. This makes one-type-per-tick viable: a reads-tick hits Processing and consumes budget; a writes-tick hits Forwarding and consumes budget; the cap that doesn't match just doesn't run. To force "Server can't handle Wave 3 lone", reduce the pooled budget below wave intensity (rather than trying to per-type-limit caps).
- **TD capacity re-tune (Stage 3c one-type-per-tick):** Server Processing `throughputPerTier: 15` (was 20), Server Forwarding `throughputPerTier: 15` (was 12), Storage `throughputPerTier: 50` (was 25). Server pooled budget = 30, DB writes sink = 50. Wave 1 (10/tick) â‰¤ 30 âœ“, Wave 2 (25/tick single-type) â‰¤ 30 âœ“, Wave 3 (50/tick single-type) > 30 â†’ lone Server loses cleanly. Cross-tick buffering (overflow stays in `state.pending`) lets Wave 3 Cache-rescue and LB-rescue both win: reads-ticks use the pooled budget for any cache misses, writes-ticks overflow buffers into the next reads-tick where Server has idle capacity. Sandbox caps are unchanged (different factory paths).

## Stage 3d engine contract gotchas

- **`forwarding-pipe` tier-1 throughput is 200/tick** (was 55 in Stage 3c). Raised when Stage 3d added Wave 4 (80/tick) and Wave 5 (150/tick). The engine counts every dequeued request against a component's tick budget regardless of whether it terminates via `RESPOND` (cache hit) or `FORWARD` â€” so CDN/Cache budgets must be â‰¥ wave intensity, not just â‰¥ the forward rate. If you're tuning a future wave with intensity > 200, bump this again in `register-td-defaults.ts` and mirror in `buildLoadBalancer` in `tests/integration/td/helpers.ts` (manual construction path).
- **`forwarding-pipe.handledTypes` must include every request type present in the wave's composition.** Stage 3d's retune extended it to `["api_read", "api_write", "static_asset", "auth_required"]`. A type missing from the list makes Cache/CDN/LB/Client silently refuse to forward that type â€” requests fall through `canHandle` and become silent PASS, which reads as mysteriously-low availability. Same discipline applies to `buildLoadBalancer`'s hand-constructed ForwardingCapability.
- **`Connection.bandwidth` defaults to 100 via `makeConnection` / `wire`.** Any wave with intensity > 100/tick silently loses ~(intensity - 100)/tick to connection-level drops, which **looks** like a component bottleneck but is actually a transport gate. Stage 3d's `wire()` helper in `tests/integration/td/helpers.ts` now takes an optional `{bandwidth}` param; Wave 4/5 tests pass `bandwidth: 200`. The dashboard's `TDModeController.tryConnect` still hard-codes `bandwidth: 100` â€” future waves with intensity > 100 will need a wave-level connection-bandwidth tuning or a `TDWaveDefinition.connectionBandwidth` override. **This is the next bottleneck to hit.**
- **`ProcessingCapability.typeLatencyPenalty` is decorative under throughput pressure.** The Stage 3d spec intended the `auth_required +4` latency penalty to fail the SLA latency gate for Wave 5 server-only, but in practice Server's 15/tick processing budget gets overwhelmed by 150/tick volume long before latency matters â€” availability collapses first. The latency penalty is still set on the TD Server factory and would matter in lower-intensity waves or if wave intensity drops. Don't rely on per-type latency penalties as the primary failure mode at high intensity; they're a decoration on a throughput-bound loss.
- **`AuthCapability.authProcessed` is reset per tick. `authProcessedTotal` is cumulative.** Stage 3d added `authProcessedTotal` to `AuthCapability.getStats()` because integration tests that check "did the Gateway actually terminate auth_required?" need a run-total stat, not a per-tick value that gets zeroed mid-run. The per-tick field is preserved for the monitoring HUD. Use `authProcessedTotal` in post-wave assertions.
- **`CapabilityStats.hitRateByType` is the per-type hit-rate field exposed by `CachingCapability`.** Shape: `Record<string, { hits: number; misses: number; hitRate: number }>`. Only present when the capability has processed at least one request. The loss modal's "Cache hit rates:" section in `src/dashboard/main.ts` gathers this via `gatherPerTypeCacheStats` and skips any caching component with an empty or absent map.
- **`CapabilityStats` interface is a hybrid of named-optional fields + a widened index signature.** Stage 3d promoted `hits?`, `misses?`, `size?` to named properties (so `(stats.hits ?? 0) + (stats.misses ?? 0)` typechecks) and added `hitRateByType?` alongside. The index signature is `[key: string]: number | Record<string, HitRateByTypeEntry> | undefined` to accommodate both shapes. Future cleanup: the index signature should eventually be removed in favor of a closed interface, listing every capability's ad-hoc stat fields explicitly. Not blocking.
- **Wave 4 intended rescue is `Client â†’ CDN â†’ Cache â†’ Server â†’ Database`** (4 components, not 3). Wave 4 traffic (80/tick, 40% non-static) overflows Server's 30/tick pooled budget even with CDN absorbing static_asset â€” the Cache is needed to absorb api_read hot-key volume. This mirrors how a player who won Wave 3 with Cache carries it forward into Wave 4 and adds CDN on top.
- **Wave 5 intended rescue is `Client â†’ CDN â†’ Gateway â†’ Cache â†’ LB â†’ [ServerÃ—2] â†’ Database`** (8 placed components including 2 servers). Gateway absorbs auth_required via `terminateAuthRequired: true`; LB + 2 Servers absorbs the remaining 120/tick of non-auth non-static traffic. A single Server cannot win Wave 5 even with full edge support because 150/tick Ã— 80% non-static = 120/tick and Server processing is capped at 15/tick per instance.
- **Wave 5's starting budget of $800 only works because topology persists across waves.** Dashboard wave-transition (`src/dashboard/main.ts:531-562`) preserves `state.components` on win â€” player carries Wave 4's topology into Wave 5 and only pays for the incremental components (Gateway + LB + 2nd Server â‰ˆ $525). Headless integration tests bypass this by constructing components directly, so `budget.passed` is trivially true via revenue accumulation â€” the test validates RESCUE CAPACITY, not placement affordability. Final affordability validation happens in the dashboard playtest.

## Stage 3d known-broken items (handed off to next session)

Surfaced during Stage 3d playtest and left unresolved when the branch pivoted to close out. Anyone picking up wave implementation work should review these before changing related code.

- **Hop dot stagger doesn't visually land at the default tick speed.** The adapter (`src/dashboard/render/state-to-renderer.ts`) computes per-group `spawnOffsetMs` values that should stagger downstream dots relative to upstream ones, and a unit test (`tests/unit/state-to-renderer-aggregation.test.ts`) confirms the offsets are correct (0 â†’ ~95ms â†’ ~190ms at a 720ms tick). The renderer's `spawnDotImmediate` sets `graphic.visible = false` during the stagger window and `tickFrame` flips it back on when `t >= 0`. Despite all that, playtest reports LB dots still appear simultaneously with Client dots on tick 1. Suspect list: (1) Pixi v8 batches visibility changes across frames, making the initial `visible = false` ineffective; (2) `performance.now()` drift between the adapter tick handler and the Pixi ticker; (3) the observer is actually seeing dots from a PREVIOUS tick that are mid-animation, not the new tick's dots. Next step: add a dashboard-side debug log in `spawnDotImmediate` and walk through DevTools console output tick by tick. Consider also: move the stagger into an *engine-level* notion of "hop tick" so the adapter can spawn dots across multiple render frames instead of all within one frame.
- **`static_asset` and `auth_required` dot colors are unverified.** The color map entries were added (`a78bfa` lavender, `facc15` gold) but playtest was cut short before confirming they render as expected. If a follow-up shows the wrong color, check `REQUEST_TYPE_COLORS` in `pixi-topology-renderer.ts`.
- **Wave 5 latency-tax teaching didn't land as designed.** The spec intended `auth_required` Server penalty (`typeLatencyPenalty: { auth_required: 4 }`) to fail the SLA latency gate when the player tries Server-only. In practice, Wave 5's 150/tick volume overwhelms Server throughput (15/tick per instance, pooled 30) long before latency becomes the limiting factor â€” availability collapses to ~54% and the player is told "server overwhelmed," not "auth is expensive." The integration test `wave-5-server-only-loses.test.ts` asserts on availability failure, not latency. Rescue path (Gateway + LB + 2nd Server) wins via availability recovery. The latency penalty is still wired on the Server factory and would surface at lower intensities; at 150/tick it's decorative. **If a future tuning pass wants the latency-tax teaching to be the visible lesson, lower Wave 5 intensity to ~50/tick or bump the penalty magnitude dramatically.**
- **Topology-validation dry-run is unimplemented.** Cross-wave roadmap item: nothing prevents a player from building a structurally-broken topology (e.g. `Client â†’ CDN â†’ Database`, no Server in the read path) and starting the wave. The wave runs, requests get dropped with `no_handler` at Database, the loss modal shows the generic "Too many requests were dropped" default diagnose-wave branch, and the player has no actionable signal. Fix (queued for Stage 3e): on READY, trace a synthetic request of each type in `wave.composition` from the entry point. If any type dead-ends, block simulation start with a toast.
- **Dashboard retry/replay may not restore connection bandwidth 300.** The `TDModeController.tryConnect` default was bumped from 100 â†’ 300 in Stage 3d. Action-log replay calls `tryConnect` again, so replayed topologies get the new bandwidth. But if any code path constructs `Connection` literals directly (not via `tryConnect`), those will still have whatever bandwidth the literal specifies. Grep for `bandwidth:` and audit.
- **`evaluateOutcome` budget display rounds to full cents.** The loss modal shows `budget: $2376.999999999959` (float drift). Cosmetic. Fix: `Math.round(actual * 100) / 100` in the notes builder in `evaluateOutcome`, or display-side formatting in `showLossModal`.
- **Palette filtering by `availableComponents` is done by hiding buttons with `b.hidden = true`.** Works but the hidden elements still occupy no DOM layout, so the palette shifts position as the player unlocks new components. Might look jumpy wave-to-wave. If it bothers the UI polish pass, consider rendering all palette entries statically and using `disabled + "locked"` styling instead of hiding.

## Post-3b cleanup pass gotchas

- **`TDModeController.advancePhase` is terminal-aware.** On the final wave's assessâ†’build transition, `currentWaveIndex` is bumped past the end and the phase **stays at "assess"** rather than wrapping back to build (there is no next wave). `isCampaignComplete()` becomes true. Subsequent `advancePhase` calls **throw** `"campaign complete"` â€” the dashboard must check `isCampaignComplete()` before re-entering the phase machine, and the HUD must gate palette / READY off `(isCampaignComplete() === false && phase === "build")`. Pre-cleanup, the dashboard could re-fire `advancePhase` post-victory and crash via `getCurrentWave()`.
- **Sandbox scenarios do not persist component ids.** `SandboxScenario.trafficSources` is `Omit<SandboxTrafficConfig, "targetEntryPointId">[]` because component ids drift across topology rebuilds (the dashboard's `topologies.ts` uses a module-level registry whose counter keeps incrementing on reset). `applyScenario(scenario, controller, entryPointId)` takes the live entry-point as a third argument and rebinds every traffic source on load. `exportScenario` strips the field. The dashboard call site passes `topo.entryPointId`. Pre-cleanup, saved scenarios silently black-holed traffic after a topology switch because reapplied stale ids referenced no live component.
- **Sandbox tryPlace/tryUpgrade and TD tryUpgrade THROW until Stage 3c.** They used to return `{ok: true, componentId: "sandbox-placed-N"}` or `{ok: true, newPlayerTier: tier+1}` without mutating state. Any future UI code that trusts the result will throw loudly instead of silently desyncing. The 4 sandbox tests that codified the lie were replaced with throw-assertions; TD `tryUpgrade` had no real test coverage to update.
- **`tests/harness/td-fixtures.ts` is the canonical TD test boot.** Exports `makeRng(seed)`, `bootTDRegistry()`, and `makeTDController(opts)`. The 6-line LCG and the `CapabilityRegistry â†’ ComponentRegistry â†’ registerTDDefaults â†’ TDEconomy â†’ TDModeController` boilerplate were hand-rolled in 5+ test files pre-cleanup. `tests/integration/td/helpers.ts` re-exports `makeRng` for any callers that still import it from there.
- **`SandboxModeController.advancePhase` no longer cycles.** The post-cleanup contract for sandbox is that `initTopology` already advances buildâ†’simulate, so the scenario load handler must NOT call `advancePhase` again (pre-cleanup it landed in "assess"). Sandbox's phase state is not currently load-bearing on much UI but the bug was real.
- **Cold-review verify before deletion.** The Batch 2 cleanup tried to delete `EngineSteps` DI plumbing on a reviewer claim of "zero consumers". `engine-tick-ordering.test.ts` uses `stepsOverride` to pin the 12-step contract â€” a real consumer the reviewer missed. Lesson: always grep for consumers yourself before nuking, even when a reviewer claims none. Skipped without harm.
